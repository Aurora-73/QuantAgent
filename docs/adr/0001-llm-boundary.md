# ADR-0001: LLM 边界设计

> 定义 LLM 在系统中的角色和边界。

---

## 状态

**Accepted**

### 上下文

在设计量化交易系统时，需要确定 LLM 的定位和使用方式。市场上存在两种极端：
1. 完全依赖 LLM 做交易决策（如 TradingGPT）
2. 完全不使用 LLM，纯传统量化

本项目需要找到一个合理的平衡点。

### 可选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **方案A：LLM直接决策** | 端到端，简单 | 不可控，风险高，难以回测 |
| **方案B：LLM辅助研究** | 可控，可验证，风险低 | 需要额外的决策引擎 |
| **方案C：混合模式** | 兼顾效率和可控 | 架构复杂，需要明确边界 |

### 最终选择

**方案B：LLM辅助研究，不直接决策**

### 理由

1. **风险控制优先**：量化交易的核心是风险控制，LLM 的输出具有不确定性，不适合直接作为交易信号
2. **可验证性**：传统量化引擎的输出可以回测和验证，LLM 的决策难以复现和验证
3. **监管要求**：直接使用 LLM 做交易决策可能不符合监管要求
4. **行业最佳实践**：参考 RD-Agent、TradingAgents 等项目，LLM 更适合作为研究助手

### LLM 的正确位置

| 适合的任务 | 不适合的任务 |
|-----------|-------------|
| 技术面分析 | 直接决定买卖 |
| 基本面分析 | 直接生成仓位 |
| 情绪面分析 | 替代风控规则 |
| 新闻分析 | 替代组合优化器 |
| 多空辩论 | 替代执行引擎 |
| 风控辩论 | 无约束的自由决策 |

**一句话：LLM 是研究员，不是交易员。**

### 影响范围

- `llm/` 模块：仅包含信息处理功能
- `strategies/` 模块：不依赖 LLM 做决策
- `risk/` 模块：纯规则驱动，不使用 LLM
- `knowledge/` 模块：LLM 生成的报告仅作为参考

### 验证方法

1. 检查所有策略是否可以在 `--no-llm` 模式下正常运行
2. 检查风控引擎是否不依赖 LLM 输出
3. 验证回测结果是否可复现
