# LLM 辅助量化交易系统架构设计

> **2026-07-06 架构定位更新**：本项目定位为 **MCP Server**，将量化研究能力（行情/因子/回测/风控/知识库）暴露 MCP 工具供 Claude/Codex/外部 Agent 调用。LLM 能力由调用方提供，系统内部不内置 LLM。

> 核心哲学：传统量化引擎做交易主干，LLM 只做研究和信息处理，不直接决定下单。
> 设计目标：一个个人可实现的 Agent 基金公司
> 设计日期：2026-06-03

## 一、MCP 工具层（当前核心定位）

系统以 **MCP Server** 模式运行，通过 `mcp_server/server.py` 暴露工具，分为 3 组：

| 模块 | 工具数 | 功能 | 文件 |
|------|--------|------|------|
| Data Tools | 12 | 行情查询、历史数据、因子、指数、日历，含自动数据拉取 | `mcp_server/tools_data.py` |
| Risk Tools | 10 | 回测、压力测试、Brinson 归因、衰减检测、健康检查 | `mcp_server/tools_risk.py` |
| Knowledge Tools | 10 | 日报搜索、事件检索、Wiki 检索、决策记忆、预测准确率 | `mcp_server/tools_knowledge.py` |

其中 3 个为 **写操作工具**（`readOnlyHint: false`）：`run_backtest`、`update_data`、`run_daily_research`。

### MCP 集成方式

```
Feishu/Telegram/Slack → cc-connect → Claude Code → MCP Server → DuckDB + baostock/AKShare
```

1. 外部 Agent（Claude Code）通过 `.mcp.json` 配置发现 MCP Server
2. MCP Server 通过 stdio 子进程方式运行
3. 工具函数自动处理数据缺失：DB 无数据时自动从 baostock/AKShare 拉取

### 配置方法

`.mcp.json` 位于项目根目录：

```json
{
  "mcpServers": {
    "QuantSystem": {
      "command": "/path/to/.venv/bin/python",
      "args": ["-m", "mcp_server.server"],
      "cwd": "/path/to/quant-system/QuantAgent",
      "env": {
        "PYTHONPATH": "/path/to/quant-system/QuantAgent",
        "PYTHONIOENCODING": "utf-8"
      }
    }
  }
}
```

查看全部工具：
```bash
cd QuantAgent && source .venv/bin/activate
python -m mcp_server.server --list-tools
```

## 二、四层架构详细设计

### 第一层：数据层 (DATA LAYER)

**目标**：做扎实的数据基座，为 AI 提供高质量输入

#### 数据源

```
数据层
├── 行情数据
│   ├── 实时行情 (CTP / IB / 聚合数据)
│   ├── 历史K线 (日线/分钟线/tick)
│   └── Level2 逐笔 (如有)
│
├── 基本面数据
│   ├── 财务报表 (季报/年报)
│   ├── 估值指标 (PE/PB/PS/EV)
│   ├── 分析师一致预期
│   └── 股东变动/大宗交易
│
├── 另类数据
│   ├── 新闻资讯 (财联社/东方财富/Reuters)
│   ├── 公告信息 (交易所公告)
│   ├── 研报摘要 (万得/东方财富)
│   ├── 社交媒体 (雪球/股吧/Twitter)
│   └── 行业数据 (产业链/供需/价格)
│
├── 宏观数据
│   ├── 利率/汇率/国债收益率
│   ├── PMI/CPI/PPI/M2
│   ├── 央行政策/政府工作报告
│   └── 海外市场联动
│
└── 资金数据
    ├── 北向资金流向
    ├── 融资融券余额
    ├── ETF 份额变动
    └── 主力资金流向
```

#### AI 在数据层的职责

**不做**：让 AI 预测价格
**做**：让 AI 做数据清洗、结构化、实体对齐

```python
# 示例：公告事件结构化抽取
{
    "source": "上交所公告",
    "ticker": "600519.SS",
    "company": "贵州茅台",
    "timestamp": "2026-06-03 18:30:00",
    "event_type": "业绩预增",           # 事件类型分类
    "event_detail": {
        "metric": "归母净利润",
        "direction": "预增",
        "range": "15%-20%",
        "period": "2026H1"
    },
    "impact_objects": ["600519.SS", "白酒板块"],  # 影响对象
    "time_window": "短期1-3个交易日",              # 时间窗口
    "confidence": 0.85,                           # 置信度
    "tradability": "高",                          # 可交易性
    "sentiment": "positive",
    "similar_historical": ["2025-07-15 茅台预增18%"]  # 相似历史
}
```

#### 工具映射

| 功能 | 推荐工具 | 说明 |
|------|---------|------|
| 行情数据 | **vnpy Gateway** / AKShare | vnpy 的 CTP/IB 接口 |
| 历史数据 | **Qlib** 数据层 | Qlib 的数据处理 pipeline |
| 新闻获取 | **OpenBB** / 自建爬虫 | OpenBB 的多数据源适配 |
| 文本抽取 | LLM API (GPT/Claude/Qwen) | 结构化事件抽取 |
| 数据存储 | DuckDB / SQLite / Parquet | 轻量级列式存储 |

### 第二层：研究层 (RESEARCH LAYER)

**目标**：AI 做研究助理，降低"找想法"的成本

#### 研究工作流

```
研究层
├── 文档处理
│   ├── 研报自动摘要 (5000字 → 500字)
│   ├── 财报要点提取 (营收/利润/现金流/指引)
│   ├── 政策文件解读 (哪些行业受益/受损)
│   └── 电话会议纪要生成
│
├── 因子研究
│   ├── 因子假设生成 (LLM brainstorm)
│   ├── 因子代码生成 (自然语言 → Python)
│   ├── 因子回测评估 (配合 Qlib/VectorBT)
│   └── 因子归因分析 (为什么有效/失效)
│
├── 市场研究
│   ├── Regime 识别 (牛市/熊市/震荡/转折)
│   ├── 板块轮动分析
│   ├── 相似历史事件检索
│   └── 资金流向解读
│
└── 策略研究
    ├── 策略思路生成
    ├── 策略代码生成与重构
    ├── 回测结果解读
    └── 多策略比较分析
```

#### Agent 设计：研究助理

```python
# 研究助理 Prompt 模板
RESEARCH_ASSISTANT_PROMPT = """
你是一位量化研究助理。你的任务是帮助研究员提高效率，而不是做交易决策。

你的能力：
1. 阅读研报并提取关键信息
2. 根据市场数据提出因子假设
3. 生成因子计算代码

5. 检索相似历史事件

1. 直接推荐买卖

```

| 回测评估 | **VectorBT** + Qlib | 快速回测验证 |
| 向量检索 | ChromaDB / Milvus | 相似事件检索 |

### 第三层：策略层 (STRATEGY LAYER)

**目标**：传统方法为主，AI 为辅。AI 理解信息，量化模型执行决策。

#### 策略架构

```
策略层
├── 信号生成
│   ├── 传统因子信号
│   │   ├── 价格因子 (动量/反转/均线)
│   │   ├── 成交量因子 (量比/换手率)
│   │   ├── 波动率因子 (ATR/历史波动率)
│   │   ├── 基本面因子 (ROE/营收增速)
│   │   └── 质量因子 (毛利率/现金流)
│   │
│   └── AI 增强信号
│       ├── 新闻情绪评分 (LLM)
│       ├── 事件冲击评分 (LLM)
│       ├── 市场 regime 判别 (LLM)
│       └── 风险过滤信号 (LLM)
│
├── 信号融合
│   ├── 因子加权融合 (IC加权/机器学习)
│   ├── 传统因子权重: 70-80%
│   └── AI 信号权重: 20-30%
│
├── 仓位管理
│   ├── 组合优化 (Riskfolio-Lib)
│   ├── 风险平价 / 均值方差
│   ├── 行业/风格约束
│   └── 仓位上限/集中度约束
│
└── 交易计划
    ├── 目标持仓
    ├── 交易清单
    ├── 执行时间窗口
    └── 预期冲击成本
```

#### 关键设计原则

```
┌─────────────────────────────────────────────────────────┐
│                    信号融合原则                           │
│                                                         │
│   传统因子 (主体)          AI 信号 (辅助)                │
│   ┌──────────────┐        ┌──────────────┐             │
│   │ 动量         │        │ 新闻情绪     │             │
│   │ 反转         │   +    │ 事件冲击     │             │
│   │ 质量         │        │ regime 判别  │             │
│   │ 估值         │        │ 风险过滤     │             │
│   │ 波动率       │        │              │             │
│   └──────┬───────┘        └──────┬───────┘             │
│          │    70-80%              │   20-30%            │
│          └───────────┬───────────┘                     │
│                      ▼                                  │
│              ┌──────────────┐                          │
│              │  融合信号     │                          │
│              │  → 仓位决策   │                          │
│              └──────────────┘                          │
│                                                         │
│   原则：AI 信号只能调制(modulate)，不能主导(dominant)      │
│   AI 可以降低仓位，但不能独自决定重仓                      │
└─────────────────────────────────────────────────────────┘
```

#### 工具映射

| 功能 | 推荐工具 | 说明 |
|------|---------|------|
| 因子计算 | **Qlib** | 因子定义与批量计算 |
| 组合优化 | **Riskfolio-Lib** | 多种优化模型 |
| 快速回测 | **VectorBT** | 向量化极速回测 |
| 生产回测 | **Lean** / **vnpy** | 事件驱动精确回测 |
| 仓位管理 | 自建模块 | 结合 Riskfolio-Lib |

### 第四层：风控与执行层 (RISK & EXECUTION LAYER)

**目标**：回撤控制 > 预测准确率。这层决定长期生存。

#### 风控体系

```
风控层
├── 事前风控 (交易前)
│   ├── 单票持仓上限 (如 5%)
│   ├── 行业集中度上限 (如 20%)
│   ├── 换手率限制 (日换手 < 10%)
│   ├── 新股/ST/停牌过滤
│   └── 流动性筛选 (日均成交额 > 5000万)
│
├── 事中风控 (交易中)
│   ├── 实时 P&L 监控
│   ├── 回撤阈值预警 (日内 -2%, 累计 -5%)
│   ├── 异常波动识别
│   ├── 新闻冲击评估 (AI)
│   └── 流动性与滑点监控
│
├── 事后风控 (盘后)
│   ├── 回测 vs 实盘偏差分析
│   ├── 因子收益归因
│   ├── 交易成本分析 (佣金+滑点+冲击)
│   ├── 过拟合检测
│   └── 策略健康度评估
│
└── AI 辅助风控
    ├── 异常波动原因分析
    ├── 新闻冲击是否值得交易
    ├── 策略失效预警 (regime 变化)
    └── 回测归因与经验总结
```

#### 执行引擎

```
执行层
├── 订单管理
│   ├── TWAP/VWAP 算法拆单
│   ├── 冰山单 / 限价单策略
│   ├── 滑点控制
│   └── 成交回报处理
│
├── 券商对接
│   ├── 期货: vnpy CTP Gateway
│   ├── 股票: XTP / 恒生 / 东方财富
│   ├── 海外: IB Gateway
│   └── 数字货币: Binance / OKX
│
└── 监控告警
    ├── 策略运行状态
    ├── 网络连接状态
    ├── 异常订单检测
    └── 钉钉/微信/邮件通知
```

#### 工具映射

| 功能 | 推荐工具 | 说明 |

## 三、层级记忆系统 (Hierarchical Memory)

这是整个系统的灵魂。解决 LLM 长期记忆不足的核心问题。

### 记忆层级

```
时间尺度        文件格式              信息压缩比      生成频率
─────────────────────────────────────────────────────────────
原始数据        tick/分钟线/新闻       1:1            实时
    ↓
日报 (Daily)    daily/YYYY-MM-DD.md   100:1          每日收盘后
    ↓
周报 (Weekly)   weekly/weekNN.md      5:1            每周五
    ↓
月报 (Monthly)  monthly/YYYY-MM.md    4:1            每月末
    ↓
季报 (Quarter)  quarterly/QN-YYYY.md  3:1            每季末
    ↓
年报 (Annual)   annual/YYYY.md        4:1            每年末
─────────────────────────────────────────────────────────────
365天原始数据    →    1份年报
可能 10GB+              可能 50KB
却保留了绝大部分重要决策信息
```

## 九、风险提示

1. **这不是印钞机**: 任何量化策略都有失效的可能，AI 也不能保证盈利
2. **先模拟再实盘**: 至少跑 3-6 个月模拟盘，验证策略稳定性
3. **小资金试错**: 实盘初期用极小资金，逐步验证
4. **持续迭代**: 市场在变，策略也要跟着变
5. **不过度依赖 AI**: AI 是工具，不是圣杯。传统因子的稳定性 > AI 的花哨
6. **回撤控制第一**: 宁可少赚，不能大亏。活着比赚钱重要

---

> 最终目标：不是做一个"AI 炒股机器人"，而是做一个"AI 辅助的量化研究与交易平台"。
> AI 负责理解信息、提高效率；量化模型负责执行决策、控制风险。
> 两者各司其职，才是最稳的路线。
